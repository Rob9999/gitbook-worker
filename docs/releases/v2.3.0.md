# v2.3.0 „Pfadtreu" / "Path-True"

**Release-Datum / Release Date:** 8. Februar / February 8, 2026
**Tag:** `v2.3.0`

---

## 🇩🇪 Deutsch

### Zusammenfassung

Version 2.3.0 „Pfadtreu" löst den kritischen Customer-Bug, bei dem der
GitBook-Rename-Schritt (`gitbook_style.py rename`) die Dateipfade zerstört hat,
wenn Einzeldateien (`source_type: file`) publiziert werden.

### Änderungen

#### 🛡️ GitBook-Rename Auto-Detect & Expliziter Key

- **Auto-Detect**: Wenn **alle** Publish-Einträge im Manifest `source_type: file`
  sind, wird der Rename-Schritt automatisch übersprungen. Dadurch bleiben
  Verzeichnis- und Dateinamen exakt so erhalten, wie sie in `publish.yml`
  deklariert sind.
- **Neuer Top-Level-Key `gitbook_rename`**: Kann auf `false` gesetzt werden,
  um den Rename-Schritt explizit zu deaktivieren — auch bei gemischten
  Einträgen. Beispiel:

  ```yaml
  gitbook_rename: false
  ```

- **Priorität**: CLI-Flag `--no-gitbook-rename` > Manifest-Key `gitbook_rename` > Auto-Detect.

#### 📦 Manifest-Version 0.1.1

- `_MANIFEST_VERSION_CURRENT` im Publisher von `(0, 1, 0)` auf `(0, 1, 1)`
  aktualisiert. Kunden, deren `publish.yml` bereits `version: 0.1.1` enthält,
  sehen keine „neuer als die getestete"-Warnung mehr.

#### 📖 FAQ-Dokument

- Neues [docs/FAQs.md](../FAQs.md) mit 5 häufig gestellten Fragen aus
  realen Kunden-Debugging-Sessions:
  1. FileNotFoundError nach GitBook-Rename
  2. `project.license` fehlt
  3. Profil-Fallback auf `default`
  4. Manifest-Version-Warnung
  5. Sonderzeichen in Dateinamen

#### 🧪 Tests

- 7 neue Unit-Tests für `_should_skip_rename()` in `test_pipeline.py`:
  - Expliziter `gitbook_rename: false/true`
  - Auto-Detect bei allen `source_type: file`
  - Gemischte Einträge → kein Skip
  - Leere Publish-Liste → kein Skip
  - Expliziter Key überschreibt Auto-Detect
- **Teststand**: 459 bestanden, 11 übersprungen, 0 Fehler.

## 🇬🇧 English

### Summary

Version 2.3.0 "Pfadtreu" (Path-True) fixes the critical customer bug where the
GitBook rename step (`gitbook_style.py rename`) destroyed file paths when
publishing individual files (`source_type: file`).

### Changes

#### 🛡️ GitBook Rename Auto-Detect & Explicit Key

- **Auto-Detect**: When **all** publish entries in the manifest have
  `source_type: file`, the rename step is automatically skipped. This
  preserves directory and file names exactly as declared in `publish.yml`.
- **New top-level key `gitbook_rename`**: Can be set to `false` to explicitly
  disable the rename step — even with mixed entries. Example:

  ```yaml
  gitbook_rename: false
  ```

- **Priority**: CLI flag `--no-gitbook-rename` > manifest key `gitbook_rename` > auto-detect.

#### 📦 Manifest Version 0.1.1

- `_MANIFEST_VERSION_CURRENT` in the publisher updated from `(0, 1, 0)` to
  `(0, 1, 1)`. Customers whose `publish.yml` already contains `version: 0.1.1`
  will no longer see the "newer than tested" warning.

#### 📖 FAQ Document

- New [docs/FAQs.md](../FAQs.md) with 5 frequently asked questions from
  real customer debugging sessions:
  1. FileNotFoundError after GitBook rename
  2. Missing `project.license`
  3. Profile fallback to `default`
  4. Manifest version warning
  5. Special characters in filenames

#### 🧪 Tests

- 7 new unit tests for `_should_skip_rename()` in `test_pipeline.py`:
  - Explicit `gitbook_rename: false/true`
  - Auto-detect with all `source_type: file`
  - Mixed entries → no skip
  - Empty publish list → no skip
  - Explicit key overrides auto-detect
- **Test status**: 459 passed, 11 skipped, 0 failures.
