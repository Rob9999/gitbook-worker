# GitBook Worker v2.6.0 Release Notes

**Release date:** May 7, 2026  
**Focus:** PDF code-fence line wrapping, sample coverage, and repository documentation cleanup.

## Summary

Version 2.6.0 makes long fenced-code sections safer for customer PDFs. The
publisher now enables `fvextra`-based wrapping for Pandoc code blocks by
default, exposes a manifest switch for opt-out cases, and keeps visual wrap
markers out of the extracted text layer so PDF QA remains readable.

This release also moves root-level inventory and customer QA material into the
engineering documentation tree, refreshes generated sample PDFs, and fixes the
sample code-fence attribute syntax so Pandoc parses filename-bearing examples
reliably.

## Added

- `pdf_options.code_block_wrap` for publish entries. The default is `true`.
- Hyphenated alias support via `code-block-wrap` for compatibility with YAML
  styles that prefer dash-separated keys.
- Unit coverage for default-on behavior, explicit opt-out, alias parsing and
  generated LaTeX header output.
- German and English advanced Markdown samples with one physically wrapped
  YAML scalar and one intentionally long code-fence stress line.

## Changed

- Pandoc/LuaLaTeX PDF headers load `fvextra` when available and recustomize
  `Highlighting` and plain `verbatim` environments with `breaklines=true` and
  `breakanywhere=true`.
- Code-fence wrap symbols are suppressed with `breaksymbolleft={}` and
  `breaksymbolsepleft=0pt`, avoiding text-extraction artifacts in QA output.
- German and English sample `publish.yml` manifests now declare
  `pdf_options.code_block_wrap: true` explicitly.
- Repository inventory and customer QA notes now live under
  `gitbook_worker/docs/` instead of the repository root.

## Fixed

- Filename-bearing sample code fences now use Pandoc-compatible fenced-code
  attributes, for example ```` ```{.python title="example.py"}````.
- Removed a temporary German sample marker before rebuilding the PDFs.

## Compatibility Notes

- Existing manifests keep the new behavior by default. To preserve older
  no-wrap PDF behavior for a specific publish entry, set:

```yaml
pdf_options:
  code_block_wrap: false
```

- The feature depends on `fvextra.sty`. If the TeX environment does not provide
  it, the generated header keeps the build compatible by skipping the optional
  wrapping setup.

## Verification Status

Release validation is in progress for the final candidate. Final test counts,
PDF font gates, package build and wheel-smoke results will be recorded before
tagging `v2.6.0`.
