# GitBook Worker v2.8.0 Release Notes

**Release date:** May 9, 2026  
**Focus:** Editorial paper strategy for complex Markdown tables.

## Summary

Version 2.8.0 turns the v2.7.0 wide-table width heuristic into a dedicated
editorial table layout strategy. Paper selection now looks beyond column count
and estimated total text width: every candidate paper is scored by expected cell
wrapping, row height, header wrapping, narrow-column pressure, and unbreakable
content overflow.

The strategy is script-aware. Long German words remain covered, but CJK,
Hangul, Kana, and other long script runs are handled as multilingual layout
risks rather than as a German-only edge case. CJK-like text is treated as wide
but character-breakable, while Latin/code/identifier tokens still contribute to
unbreakable-overflow warnings.

The LaTeX renderer now uses the same width model as the strategy for wrapped
tables. Instead of selecting a larger page and then emitting natural-width
`l/c/r` columns, wrapped Markdown pipe tables are rendered as `p{...mm}`
columns with explicit widths. CJK/Hangul/Kana table cells also receive local
LaTeX break hints so the real PDF layout matches the strategy assumption that
these scripts can break at character boundaries.

## Added

- Dedicated `gitbook_worker.tools.publishing.table_strategy` module for table
  measurement, candidate scoring, override parsing, and report output.
- `pdf_options.table_paper_strategy` configuration in `publish.yml` with
  thresholds for cell lines, header lines, row-line averages, readable column
  width, unbreakable overflow tolerance, candidate papers, and oversize policy.
- Optional JSONL table layout reports for QA and editorial review.
- Per-table source override comment:
  `<!-- gbw-table paper=a3-landscape reason="reviewed editorial layout" -->`.
- German and English sample tables with long CJK/Hangul/script-run content.

## Changed

- Markdown preprocessing delegates table paper choice to the new strategy
  module while preserving compatibility helper names for existing tests and
  callers.
- The old column-count heuristic remains a lower bound, but editorial scoring
  can escalate to the next useful paper size when smaller candidates would make
  cells too tall or columns too narrow.
- Oversized tables keep the preserve-column-heuristic fallback by default and
  emit diagnostics instead of silently choosing a paper that cannot plausibly
  solve the layout.
- Folder publishing and the main publisher now pass resolved table strategy
  options through to preprocessing.
- Wrapped table rendering now uses paragraph columns with strategy-derived
  widths, preventing the rightmost columns from being clipped on wide decision
  tables.
- Script-run cells in wrapped tables receive local break hints for CJK,
  Hangul, and Kana characters.

## Configuration Example

```yaml
pdf_options:
  table_paper_strategy:
    enabled: true
    mode: editorial
    report: jsonl
    max_cell_lines: 5
    max_header_lines: 3
    preferred_max_avg_row_lines: 2.8
    min_readable_column_width_mm: 14
    unbreakable_overflow_tolerance_mm: 2
    oversize_policy: preserve-column-heuristic
```

## Compatibility Notes

- Existing projects work without configuration changes; the default strategy is
  enabled by the preprocessing layer.
- Projects that need conservative paper choices can tune thresholds or disable
  the strategy via `pdf_options.table_paper_strategy.enabled: false`.
- Custom paper candidates are accepted through configuration for editorially
  approved special layouts.

## Known Limitations

- The strategy still estimates layout before Pandoc/LuaLaTeX rendering; exact
  TeX box measurement is intentionally out of scope for this release.
- Reports are diagnostic artifacts and are not yet rendered into a human-facing
  HTML dashboard.
- Tables that exceed all configured candidates still require editorial action:
  split the table, shorten columns, add an approved custom paper, or use an
  explicit table override.
- Font gates verify configured font embedding and positive text extraction, but
  visual replacement glyphs can still indicate coverage gaps in the configured
  CJK/Hangul sample font set and must be handled as a font inventory issue.
